线程的基本概念

线程的生命周期

线程一共有六种状态，NEW、RUNNABLE、BLOCKED、WAITING、TIME_WAITING、TERMINALED

启动一个线程前，最好为这个线程设置线程名称，因为这 样在使用 jstack 分析程序或者进行问题排查时，就会给开发人员提供一些提示

线程的启动原理

为什么调用start方法而不是run方法

线程的终止

线程的终止，并不是简单的调用stop命令去。虽然api仍然可以调用，但是和其他的线程控制方法如suspend、resume一样都是过期了的不建议使用，就拿stop来说，stop方法在结束一个线程时并不会保证线程的资源正常释放，因此会导致程序可能出现一些不确定的状态。要优雅的去中断一个线程，在线程中提供了一个interrupt方法
wait()、sleep()、Thread.join()为什么会抛出InterruptedException

需要注意的是，InterruptedException 异常的抛出并不意味着线程必须终止，而是提醒当前线程有中断的操作发生， 至于接下来怎么处理取决于线程本身，比如

  1. 直接捕获异常不做任何处理
  2. 将异常往外抛出
  3. 停止当前线程，并打印异常信息

synchronized 有三种方式来加锁
对象在内存中可以分为3部分，对象头、实例数据、对齐填充三部分组成

为什么任何对象都可以实现锁

Synchroized的锁升级

synchronized 在 JDK1.6 之后做了一些优化，为了减少获得锁和释放锁带来 的性能开销，引入了偏向锁、轻量级锁

偏向锁的获取

在我们的应用开发中，绝大部分情况下一定会存在 2 个以 上的线程竞争，那么如果开启偏向锁，反而会提升获取锁 的资源消耗。所以可以通过 jvm 参数UseBiasedLocking 来设置开启或关闭偏向锁

轻量级锁的升级过程

轻量级锁升级后mark word也会进行相应的改变
  1. 线程在自己的栈帧中创建锁记录，LockRecord
  2. 将对象头中的mark word复制到锁记录中
  3. 将锁记录的指针指向锁对象
  4. 将锁对象的对象头的 MarkWord 替换为指向锁记录的指针

自旋锁

  1. 线程原地循环，不是释放cpu，而是消耗cpu的循环
  2. 适合同步代码执行时间很短的场景，看似无意义的循环，反而能提升锁的性能
  3. 自旋需要条件控制，如果同步代码执行时间过长会给服务带来性能问题，默认自旋10次，次数可以通过preBlockSpin调节
jdk1.6引入了自适应的自旋锁

重量级锁的基本原理

当轻量级锁膨胀到重量级锁时，线程只能被置于阻塞状态
monitor 依赖操作系统的 MutexLock(互斥锁)来实现的, 线程被阻塞后便进入内核(Linux)调度状态，这个会导致系统在用户态与内核态之间来回切换，严重影响锁的性能

wait和notify

wait()：表示持有锁的对象准备释放锁，并进入等待状态，进入等待队列(处于WAITING状态)中

notify():表示唤醒等待队列中的线程，并把线程放入到同步等待队列(BLOCKED状态)中

volatile

volatile的作用
volatile就是用来保证跨线程写入的内存可见性问题

CPU缓存一致性问题

CPU的执行速度非常快，但是总体的执行效率依赖于最慢的那个，基于这个情况，CPU做一些优化，其中就有CPU高速缓存，CPU高速缓存分为3级，一级缓存、二级缓存、三级缓存，这样很好的解决了速度的矛盾，但是增加了系统的复杂度，从而产生了缓存一致性的问题

  • 总线锁

    在其中一个CPU操作共享内存时，发出lock指令，锁住共享内存区域，这样就解决了一致性问题，但是开销很大，不适合此场景

  • 缓存锁

    只需要保证多个CPU缓存数据的一致性，MESI协议
    MESI 表示缓存行的四种状态，分别是
    1. M(Modify) 表示共享数据只缓存在当前 CPU 缓存中， 并且是被修改状态，也就是缓存的数据和主内存中的数据不一致
    2. E(Exclusive) 表示缓存的独占状态，数据只缓存在当前 CPU 缓存中，并且没有被修改
    3. S(Shared) 表示数据可能被多个 CPU 缓存，并且各个缓存中的数据和主内存数据一致
    4. I(Invalid) 表示缓存已经失效

    CPU读请求，可以读取处于M、E、S状态的缓存

    CPU写请求，可以读取M、E状态的缓存，如果想修改S状态的缓存，需要把其他CPU中的缓存修改为I

    MESI 优化带来的可见性问题

    因为当CPU在缓存中对共享变量进行写入时，需发送让消息失效的请求给其他CPU，这样当前线程是处于阻塞状态的，这样会带来资源的浪费，所以在CPU中又引入了storeBufferes

    引入了storeBufferes后产生两个问题

    • 数据什么时候提交是不确定的，因为需要等待其他 cpu给回复才会进行数据同步。这里其实是一个异步操作
    • 引入storeBUfferes后，处理器会优先读取storeBufferes中的值，否则再从缓存行中读取
    当执行完修改后，会把数据写入到storeBufferes中，之后继续向下执行，这样就会产生指令重排序的问题，进而可能导致可见性问题，从而引入了内存屏障的概念

    内存屏障

    强制把storeBufferes中的数据刷新到主存中

    最后得出的结论为导致可见性问题的根本就是缓存和指令重排序的问题，JMM采取合理的禁用缓存、禁用指令重排序的方式，来根本解决可见性和有序性的问题

    采用volatile修饰字段，就是在读写字段前加入一些内存屏障，这些屏障要求编译器适当的禁止指令重排序，要求处理器刷新缓存

HappenBefore

为了保证内存可见性，Java 编译器在生成指令序列的适当位置会插入内存屏障来禁止特定类型的处理器的重排序
五种happenBefore

volatile总结

为了保证可见性，采用volatile修饰的变量，在读取和写入变量的前后都会加上内存屏障，这样会使得编译器和处理器禁止指令重排序，并且不会存入storeBufferes中，会直接写入主存中，其他线程读取共享变量时也直接从主存中读取

ReentranLock底层原理

Condition

场景：线程间协调通信，await()和signal()，生产者消费者模型

CountDownLatch

场景：倒计时的概念，当count为0时，所有线程进行唤醒，FIFO队列中上一个节点唤醒下一个节点

Semaphore

场景：限流器的作用，允许最多多少个线程同时运行

CyclicBarried

场景：当存在需要所有的子任务都完成时，才执去主任务，这个时候就可以选择使用CyclicBarrier

Synchroized和ReentranLock的区别

  1. Synchroized的实现基于JVM层面，对于开发者是透明的；ReentranLock是基于JavaAPI实现
  2. Synchroized等待中的线程不可被中断；ReentranLock可被中断
  3. Synchroized只能是非公平锁；ReentranLock可实现公平锁
  4. Synchroized的wait和notify只能维护一个单向链表用来维护等待队列，具体的唤醒由JVM决定；ReentranLock的Condition可以有选择的维护多个Condition也就是多个队列，这样在调度上更加灵活，可以有选择的通知唤醒

计数流程

采用分片的思想

  1. 如果数组为空，则改变标识位，之后进行数组大小为2的初始化顺便赋值
  2. 如果数组不为空，但是数组中随机某一位为空，则修改标识位防止并发对数组进行操作，这时进行+1操作
  3. 如果数据某一位也不为空，说明没有空数组位，则直接进行cas累加数组中的其中一位
  4. 如果直接进行cas累加也失败，说明并发严重，容量不够大，此时进行扩容，先修改标志再进行扩容，防止出现并发问题，扩容方式为向左移动一位，增大2倍，更新随机数，之后再次进入循环

put流程

  1. 首先判断数组是否为空，为空则初始化数组，若传入指定大小则初始化，默认16
  2. (n-1) & hash之后获取指定位置值，若值为空则通过CAS方式插入
  3. 判断是否正在扩容，正在扩容则帮助扩容
  4. 锁住头节点，判断当前是链表还是红黑树，按指定规则插入内容
  5. 判断链表长度是否大于8并且数组长度是否大于64，如果小于64则进行数组长度扩容，否则把链表转化为红黑树
  6. 通过分片的思想记录size长度

扩容流程

  1. 采用多线程扩容的思想，如果大小达到长度的0.75倍则触发扩容机制
  2. 生成全局唯一扩容戳，判断是否正在扩容，如果正在扩容则判断任务是否被分发完毕，如果分发完毕则退出，否则参与扩容，并将扩容戳+1，代表当前有线程参与扩容
  3. 创建一个大小为原大小2倍的数组，进行多线程扩容，根据size/8/cpu核心数与16比较，分配每个线程的扩容范围
  4. 进行逐个移动，移动后的位置变成Forward对象代表已经扩容完毕
  5. 如果有链表则进行进行hash&n把结果等于0和不为0分为两个高低链，把高链放到i+n出，因为数组扩容后，hash&n，n增大，会导致内容重新排列，之后继续向前扩容
  6. 最后线程完成扩容，扩容戳-1

阻塞队列

场景：

线程池

优势
  1. 降低创建线程和销毁线程的开销
  2. 提高响应速度，当有新任务需要执行时，不需要等待线程创建就可以立马创建出来
  3. 合理创建线程池大小可以避免因为线程数超过硬件资源瓶颈而带来的问题
线程池类型

  • newFixedThreadPool

    返回一个固定线程数的线程池，若线程池中空闲则立即执行，若没有则放入到一个暂缓执行的任务队列中

    用途:FixedThreadPool 用于负载比较大的服务器，为了资源的合理利用，需要限制当前线 程数量

  • newSingleThreadExecutor

    返回一个线程数的线程池，若线程池空闲则立即执行，若没有则放入到一个暂缓执行的任务队列中

  • newCachedThreadPool

    返回一个可根据实际情况调整的线程池，不限制最大线程数，如有空闲的线程则执行任务，若无任务则不创建线程，每一个空闲线程会在60s被回收

  • newScheduledThreadPool

    创建一个可以指定线程数量的线程池，但是这个线程池还带有延迟和周期性执行任务的功能，类似定时器

线程池参数列表
线程池执行流程

  1. 判断线程池中工作的核心线程数是否小于设置的核心线程数，如果小于则创建一个线程执行任务
  2. 如果等于则把任务当到阻塞队列中，等待被处理
  3. 如果任务过多，导致阻塞队列已满则判断当前工作线程是否大于最大线程数，如果小于最大线程数则创建一个线程执行任务
  4. 如果工作线程已与最大线程一致则执行饱和策略
线程池中的几种状态

  • RUNNING 运行状态，接收任务，处理阻塞队列中的任务

  • SHUTDOWN 关闭状态，不接受新任务，只处理阻塞队列中的任务

  • STOP 停止状态 不接受新任务，不处理阻塞队列中的任务，中断正在执行的任务

  • TIDYING 结束状态 所有任务都已结束，线程数量为0

  • TERMINATED 销毁状态，一切结束

拒绝策略
  • AbortPolicy:直接抛出异常，默认策略;
  • CallerRunsPolicy:用调用者所在的线程来执行任务;
  • DiscardOldestPolicy:丢弃阻塞队列中靠最前的任务，并执行当前任务;
  • DiscardPolicy:直接丢弃任务;
  • 或者自定义饱和策略，可以采取持久化等操作

如何合理配置线程池的大小

分析应用是CPU密集型还是IO密集型
  1. 如果是CPU密集型，CPU利用性很高，CPU一直在工作，这样线程数最好设置为CPU核心数+1，设置过高可能还会因为频繁切换执行线程导致性能下降
  2. 如果是IO密集型，CPU会有很长时间的空闲时间，这样可以把线程数设置为CPU核心数*2

线程池设定最佳线程数目 = (线程池设定的线程等待时间+线程 CPU 时间)/ 线程 CPU 时间 )* CPU 数目

Callable和Future的使用

execute 和 submit 区别
  1. execute 只可以接收一个 Runnable 的参数
  2. execute 如果出现异常会抛出
  3. execute 没有返回值

死锁

预防死锁
  1. 所有线程按照固定顺序获取锁
  2. 使用Lock尝试获取锁，设置超时时间，超时放弃
解除死锁
  1. 剥离资源，达到解除死锁
  2. 撤销进程

ThreadLocal

原理

创建每个线程独有的一份部分内容

场景

用于登陆的token，每个线程独有自己的token信息

有可能导致内存泄漏

threadLocal里面的key是弱引用，value是强引用，会出现把key置为null，但是value为强引用不回收的情况，最终导致内存泄漏

解决方案
  • 每次使用完ThreadLocal，都调用它的remove()方法，清除数据
  • 在使用线程池的情况下，没有及时清理ThreadLocal，不仅是内存泄漏的问题，更严重的是可能导致业务逻辑出现问题。所以，使用ThreadLocal就跟加锁完要解锁一样，用完就清。
